# 2026-06 — The kiosk, as a real service

**Goal:** the small node that used to run the dashboards — now relieved of them and
reflashed to the **lite, headless OS** — still drives a little 7" touchscreen on the
wall. The old kiosk was just a browser someone opened by hand on the full desktop.
With the desktop gone, rebuild the kiosk as a **managed, resilient, observable
service**, not a hand-started window.

## The decision: a service, not an autologin shell

There are two ways to put a fullscreen browser on a headless lite box: an **autologin**
on a virtual terminal whose shell launches a compositor, or a **systemd service**.

I chose the service — *knowing* it costs more up front. A headless service has no login
seat by default, so the compositor can't become the display's owner or grab the
touch input until you wire it a session. The autologin path gets that seat for free.

But the service buys what actually matters for an unattended wall display:

- **restart-on-crash** — the display heals itself;
- **logs to the central store** — the kiosk's output ships to the same log hub as the
  rest of the fleet, instead of vanishing onto a terminal;
- **start-after-network** — it opens the board only once the network and the gateway's
  proxy are reachable, so a cold boot doesn't flash a "can't connect" error.

The clincher: this node is **useless if it's off the network** anyway, so the autologin
path's headline advantage — poking at it on the local console — buys nothing. If it's
reachable I'll debug it remotely; if it isn't, a login shell on the screen won't help.
Everything's remote now regardless.

## What got built

A minimal Wayland **kiosk compositor** running a **single fullscreen browser** at the
fleet-overview board, as a systemd unit (restart always, after-network-online, output
to the journal → central log store).

Two details worth noting:

- **The browser trusts the fleet CA through its *own* store.** The internal dashboard
  URL is HTTPS signed by the homelab's private CA, which the system already trusts — but
  the browser keeps its **own** certificate store and ignores the system one. Without
  loading the CA there too, the kiosk lands on an un-clickable certificate warning. So
  the role installs the CA into the browser's store explicitly.
- **The kiosk is now observable.** A unit-scoped metric exports whether the kiosk
  service is alive, and an alert fires when the **box is up but the display is dead** —
  the failure the generic node-down alert can't see, because the node itself is fine.

## Two things fought back — one fixed, one parked

**1 — the dashboard wanted a login (fixed).** The board first loaded… to a login page: the
kiosk has no user and can't type credentials, so it just sat there. The fix was to let the
dashboard server serve a **read-only anonymous view** (LAN-only, behind the gateway's proxy —
fitting the homelab's "favor visibility on the trusted LAN" posture), rather than give the
kiosk a login to babysit. One paired gotcha: the kiosk-mode flag rides on the URL's query
string, and pointing at the dashboard's *bare* id made the server 302-redirect to append the
human-readable slug — **dropping the query** in the process, so the board came up in normal,
chrome-on mode. Launching the full id-plus-slug URL keeps the flag intact. The board now
draws straight to the wall: no login, no chrome.

**2 — the scale knob scaled the wrong thing (parked).** The 7" panel is tiny, so the board
falls back to a cramped single-column layout with ballooning text. The obvious fix — force a
smaller *device scale factor* in the browser — didn't reflow the content: it shrank the
**whole window** into ~60 % of the screen, leaving a dead border around a smaller picture of
the same cramped layout. I wanted **denser content filling the screen**, not a smaller copy
of it. So the lever is wrong — under this compositor the browser's device-scale-factor behaves
like an *output* scale. The right lever is something else: a larger logical resolution, an
output-level scale, or page-level zoom. Still open, parked for now.

## What I'd tell past me (so far)

- A "kiosk" on a headless box is 10 % browser and 90 % **seat/session plumbing**. Budget
  for it.
- "Make it a proper service" pays **compounding interest**: the same unit that restarts
  the display also makes it observable and alertable, almost for free.
- **Verify the actual pixels.** Both open issues only appeared on the real screen — a
  login wall and a mis-scaled board are both invisible from the command line.

## Addendum — 2026-06-24: placing the parked lever (a realignment, not a rewrite)

Coming back to this with sharper vocabulary, the parked scale issue (§2) is worth classifying
precisely. The text above stays exactly as written — this only adds; the amendment lives in the
log, while the present-state notes carry only where the kiosk *is* now.

The scale-factor choice was **a deliberate, reasonable lever — not a defect, and not a
should-have-known mistake.**

- It isn't a **bug.** Nothing executed incorrectly; the browser did precisely what that flag
  instructs. A bug would be a sound design whose *implementation* misbehaves — this is the
  opposite: a faithful implementation of a lever that turned out to be the wrong one.
- It isn't a **foreseeable miss** either. `--force-device-scale-factor` is, in an ordinary
  browser, a content/DPI scale, so "it densifies content" was the sound expectation going in.
  That it acts as an **output scale under this Wayland compositor** is a property you can only
  surface by *running it on the real panel* — invisible from the docs and from the command line.

So this is **an evolution on knowledge gained**: a deliberate choice with an unexpected result we
could not have falsified in advance, not a premise we should have checked. The general discipline
it follows — and the reason this is an addendum rather than an edit: **amend the historic log,
never overwrite it; let current-state/architecture statements reflect the present live network.**
The cautionary trail (the wrong-lever reasoning) stays above; the live truth is just "the
device-scale-factor lever is known-wrong under cage; the right one — larger logical resolution
vs. output-level scale vs. page zoom, and which survives reboots — is still open."

## Resolution — 2026-06-24: a single table, not a scaled grid

It's solved — but not by finding the right scaling lever. Both browser-side levers turned out to be
dead ends, and both were only knowable by trying them on the real panel:

- **device-scale-factor letterboxes.** Under this kiosk compositor a sub-1.0 device-scale-factor
  shrinks the whole window into a corner with a dead border, instead of densifying content.
- **The kiosk browser ignores page zoom outright.** Presetting a default zoom level in the browser
  profile did nothing; so did *injecting* zoom keystrokes into the live page programmatically. Locked
  fullscreen "kiosk" mode simply drops zoom.

So the fix sidesteps scaling entirely. The board only collapsed into a cramped single column because
it had **many panels** to stack below a fixed width breakpoint. Rebuilt as **one panel** — a single
table, one row per host, columns for status / CPU / memory / disk / temperature with threshold
colouring — there is nothing to stack, so it renders dense and legible at the panel's native size no
matter the width. The lesson the §2 addendum was circling becomes concrete: when the platform fights
you on *how big*, change *what you're laying out*.

Two things fell out of the chase, both keepers:

- **A remote screenshot of the live wall.** To verify layout without standing in front of the screen,
  the kiosk now exposes the browser's own debugging port (loopback only) and a small helper grabs a PNG
  of *exactly* what's on the panel over SSH. It reuses the already-running browser — no second process,
  which matters on a 1 GB node. Being able to *see* the wall remotely turned a guess-and-walk-over loop
  into a tight, head-down one.
- **Navigation back.** Dropping the URL's kiosk flag (distinct from the browser's own kiosk lock)
  restores the dashboard nav, so the touchscreen can switch between boards — the panel doubles as a
  walk-up console now, not just a fixed display.
